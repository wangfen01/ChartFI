This set of heatmaps displays the frequency of various build and compile error types across several projects and multiple Large Language Model (LLM) configurations, with darker reds indicating higher error counts after manually combining highly similar clusters and removing less relevant data such as missing files, build timeouts, and successful outputs. The error profile is highly uneven, showing that while most project–configuration combinations are near zero, specific hotspots dominate the totals, suggesting that certain projects are particularly sensitive to specific failure modes and that some configurations systematically trigger particular classes of errors. Across applications and LLMs, errors relating to CMake configuration, as well as undeclared identifiers and function argument or type mismatches for apps besides nanoXOR, are broadly common, suggesting that coordination of function interfaces and variable names across files and successful configuration of CMake for Kokkos builds are common points of difficulty across LLMs. Build-system and configuration failures show a sharp spike in Makefile Missing Build Target for nanoXOR under the Llama-3.3-70B configuration, while CMake Config Error shows repeated mid-to-high counts for microXOR and microXORh across several configurations, indicating persistent configuration-generation issues. In contrast, CMake or Makefile Syntax Error is largely absent except for a notable surge for SimpleMOC-kernel under gemini-1.5-flash and Llama-3.3-70B, implying configuration-specific templating or formatting problems. Compiler and code-level errors show distinct clustering, with Invalid Compiler Flag overwhelmingly driven by SimpleMOC-kernel, where the gemini-1.5-flash and gpt-4o-mini configurations produce large jumps relative to the rest of the matrix. Missing Header File is concentrated in XSBench, particularly under gemini-1.5-flash and gpt-4o-mini, pointing to include-path or dependency resolution problems. Undeclared Identifier is dominated by microXOR, which exhibits very high counts across multiple configurations—peaking at 75 for gemini-1.5-flash—suggesting systematic code generation issues, while Function Argument or Type Mismatch concentrates in microXORh and microXOR, indicating that type consistency and API usage degrade sharply under these setups. Linker Error is comparatively rare except for a single standout spike of 41 for microXOR under gpt-4o-mini, reflecting a potential anomaly in library linking or build artifact configuration. OpenMP Invalid Directive remains a secondary issue with low scattered counts, reinforcing the conclusion that the dominant risks differ by project: nanoXOR is most vulnerable to build-target specification, SimpleMOC-kernel to compiler-flag compatibility, XSBench to header resolution, and the XOR variants to semantic correctness in generated code.
